## 1. 현재 프로젝트 상태 요약

이 리포는 원래 `SH Homepage Renewal` 문서 구조를 가지고 시작했지만, 실제 구현은 현재 `동연 파워랭킹 + 사냥/성장형 캠퍼스 게임 서비스` 방향으로 확장되어 있다.

현재 서비스에는 이미 아래 범주가 들어와 있다.

- 로그인 / 회원가입
- 메인 로비
- 사냥터 선택 / 전투 화면
- 몬스터 / 드랍 테이블
- 인벤토리 / 장착 / 해제
- 장비 강화
- 카드
- 상점
- 컬렉션 / 도감
- 전투력 랭킹
- 미션 / 업적 / 성장 축 UI

하지만 중요한 차이는 있다.

- 일부 기능은 서버 DB 기반으로 동작한다.
- 일부 기능은 `localStorage` 기반으로만 동작한다.
- 일부 API는 실제 저장이 아니라 하드코딩/응답 stub 수준이다.

즉, 화면은 많이 나와 있지만 데이터 영속화는 아직 절반 정도만 끝난 상태다.

## 2. 구현 상태 점검

### 2.1 인증 / 유저

상태: `완료`

구현됨:

- `POST /api/users/signup`
- `POST /api/users/login`
- `GET /api/users/me`
- `POST /api/users/refresh`
- `POST /api/users/logout`

비고:

- 사용자 인증 자체는 서버 세션/토큰 기준으로 동작한다.
- 이 영역은 현재 게임 기능 중 가장 안정적으로 서버 연동된 파트다.

### 2.2 전투 시스템

상태: `부분 완료`

구현됨:

- `GET /api/zones`
- `GET /api/zones/:zoneId`
- `GET /api/zones/:zoneId/drops`
- `GET /api/combat/state`
- `POST /api/combat/click`
- `POST /api/combat/use-consumable`

구현 내용:

- 사냥터/몬스터 조회
- 클릭 공격
- 몬스터 처치
- 드랍 지급
- 다음 몬스터 스폰
- 카드 선택 효과 일부 반영
- 클릭 한도 / 보스 입장 제한 / 드랍 버프 / 공격 버프 반영

문제:

- 전투 세션 상태는 서버 메모리 기반이다.
- 서버 재시작 시 몬스터 현재 HP, 버프, 전투 로그 등은 초기화된다.
- 유저의 사냥 진행도 전체는 DB가 아니라 클라이언트 `localStorage` 쪽이 더 큰 비중을 가진다.

결론:

- 전투 API는 존재하지만 `운영형 영속 전투 시스템`은 아니다.

### 2.3 인벤토리 시스템

상태: `부분 완료`

구현됨:

- `GET /api/inventory`
- `POST /api/equipment/equip`
- `POST /api/equipment/unequip`
- `POST /api/items/use`
- `POST /api/items/sell`

구현 내용:

- 장비 인벤토리 조회
- 장착 / 해제
- 파워랭킹 소비 아이템 사용
- 판매 처리

문제:

- `GET /api/inventory`는 장비와 파워랭킹 소비 아이템 위주다.
- 사냥 재료 / 게임 소비템 / 기타 아이템 전부를 통합해서 서버에서 관리하지 않는다.
- 실제 게임형 인벤토리 핵심인 `골드 / 젬 / 강화석 / 카드 조각 / 이벤트 토큰`은 DB가 아니라 `localStorage`에 있다.

결론:

- 장비 인벤토리는 어느 정도 서버형.
- 게임 인벤토리 전체는 아직 미완성.

### 2.4 장비 시스템

상태: `부분 완료`

구현됨:

- 장비 조회
- 장착 / 해제
- 강화 미리보기
- 강화 요청
- 전투력 계산
- 6부위 슬롯
  - `weapon`
  - `head`
  - `top`
  - `bottom`
  - `shoes`
  - `gloves`

구현 내용:

- 무기 고정 공격력
- 상의/하의 퍼센트 보정
- 세트 배수
- 카드 성장 배수
- 드랍 보정 / 자동 성장 / 보스 추가 롤

문제:

- 강화 단계 저장이 DB가 아니라 `localStorage` 기반이다.
- 세트 정보는 `GET /api/collection/sets`에서 하드코딩 반환이다.
- 장비 획득 루프가 아직 `장비 30종`, 제작, 등급별 성장 구조까지는 안 갔다.

결론:

- 계산식은 상당히 들어갔지만, 영속화와 장비 풀 확장이 필요하다.

### 2.5 카드 시스템

상태: `부분 완료`

구현됨:

- `GET /api/cards`
- `GET /api/user/cards`
- `POST /api/cards/select`
- `POST /api/cards/upgrade`

구현 내용:

- 고정 카드 카탈로그 분리
- 카드 타입
  - 공격 카드
  - 수확 카드
  - 응원 카드
  - 인기 카드
- 카드 선택 시 전투/드랍/클릭/인기도 성장 보정 반영
- 카드 레벨 표시
- 카드 인기도 누적

문제:

- 카드 선택/레벨업 API는 실제 저장이 아니라 성공 응답 stub 성격이 강하다.
- 카드 레벨 / 인기도 / 선택 상태는 여전히 `localStorage` 기반이다.
- 카드 조각을 이용한 서버형 해금/성장 루프가 없다.

결론:

- 카드 기능은 “체험 가능한 수준”까지는 왔지만 서버 영속화가 필요하다.

### 2.6 상점 시스템

상태: `부분 완료`

구현됨:

- `GET /api/shop/items`
- `POST /api/shop/buy`

구현 내용:

- 소비 아이템 카탈로그 조회
- 강화 재료 / 버프 아이템 / 카드 성장형 소비템 구성

문제:

- `POST /api/shop/buy`는 상품 존재 여부만 체크하고 `{ item }`를 반환한다.
- 실제 구매 비용 차감과 아이템 지급은 프론트 `localStorage`에서 처리한다.

결론:

- 상점 목록은 있음
- 실제 서버형 구매 시스템은 아직 아님

### 2.7 컬렉션 / 도감 / 세트

상태: `부분 완료`

구현됨:

- `GET /api/collection/equipment`
- `GET /api/collection/monsters`
- `GET /api/collection/sets`

문제:

- 몬스터/장비/세트 조회는 대부분 카탈로그/하드코딩 기반이다.
- 유저별 수집 달성도, 세트 달성도, 도감 보너스 영속화는 없다.

결론:

- 읽기 전용 컬렉션 화면 수준
- 실제 성장 시스템과 완전 연결되지는 않음

### 2.8 미션 시스템

상태: `부분 완료`

구현됨:

- 메인 로비에
  - 일일 미션
  - 주간 미션
  - 업적
  - 성장 축
- 미션 보상 수령 버튼
- 보상 지급 로직

문제:

- 전부 `localStorage` 기반이다.
- 서버 API가 없다.
- 보상 수령 내역도 서버 저장이 없다.

결론:

- UI와 루프는 있음
- 운영형 미션 시스템은 아직 아님

### 2.9 자원 시스템

상태: `부분 완료`

게임 자원 종류:

- 골드
- 젬
- 강화석
- 고급 강화석
- 보호 주문서
- 제작 재료
- 카드 조각
- 이벤트 토큰

문제:

- 게임 자원은 대부분 프론트 `localStorage`에만 존재한다.
- `GET /api/user/resources`는 이름과 다르게 실제 게임 자원 응답이 아니라 파워랭킹 아이템 인벤토리 성격이다.

결론:

- 자원 체계는 정의돼 있으나 서버형 자원 시스템은 미완성

## 8. 한 줄 결론

현재 프로젝트는 `게임 플레이 가능한 데모 수준`까지는 올라왔지만,  
`운영 가능한 게임 서비스`가 되려면 다음 단계는 무조건 `DB 영속화와 API 정리`다.
